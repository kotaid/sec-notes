Why Do We Use Models?

In networking, we use models to conceptualize the network and divide it into simpler, discrete pieces — so we can understand it, troubleshoot it, and communicate about it with a shared vocabulary.

The model we rely on is the OSI (Open Systems Interconnection) model. It helps us determine which layer a problem has occurred in, and gives everyone — from hardware techs to security analysts — the same language to describe network behavior.

Network protocols are sets of rules and standards that enable hardware and software to function at a particular layer. Each protocol lives at a specific OSI layer and only concerns itself with that layer's responsibilities.

The Seven Layers

The OSI model encourages modular design — each layer has as little to do with other layers as possible.

L1 & L2 — Network Hardware Deep Dive

// Cables (L1)

The standard cabling solution is UTP (Unshielded Twisted Pairs). The twisted design reduces electromagnetic interference. Each cable consists of a solid conductor core, twisted pairs, a ripcord, and an outer jacket.

// Hubs vs Switches

Hubs repeat a received frame and send it to every connected system — inefficient and a security risk. Switches send the frame only to the intended NIC. This is why switches replaced hubs.

// NIC — Network Interface Card

The NIC is the interface between a PC and the network. Some authors place NICs in L1, others in L2 — both are reasonable.

MAC Address

Each NIC stores a MAC (Media Access Control) address burnt into its firmware ROM. Every manufacturer requests a block of globally unique MAC addresses.

00:40:05:E0:7D:A9
← OUI (Manufacturer)(Device ID) →
  • 48 bits long, represented as 12 hex characters
  • First 6 chars = OUI — assigned to the manufacturer
  • Last 6 chars = assigned by manufacturer to the specific device
  • Also called MAC48 or EUI48

NICs transmit data as electricity, light, or radio waves — binary represents these as power pulses, 1s and 0s.

Frames & Encapsulation (L2)

The NIC breaks data into chunks and encapsulates them into a frame. Each layer has a PDU (Protocol Data Unit) — at Layer 2 it's the Frame.

  • Type — Indicates what is encapsulated in this frame
  • Data — The payload; transmitted without the NIC caring about the content
  • FCS — A CRC value for data integrity. Max frame size: 1500 bytes
// How FCS WorksSending NIC calculates CRC and appends it. Receiving NIC runs the same math — match ✅ data is good, no match ❌ frame is dropped.

ARP — How Systems Find Each Other

First time a system communicates on a network, it doesn't know the target's MAC:

  1. Sends a broadcast frame to FF:FF:FF:FF:FF:FF
  2. Contains a request for the MAC address of the target IP
  3. Every system processes it — only the target responds with its MAC
  4. Future frames go directly to that MAC — no more broadcasting
// ARP = Attack SurfaceARP has no authentication — any device can respond to an ARP request. Root cause of ARP Spoofing / Poisoning attacks.

SOC Analyst Relevance

ARP Spoofing / Poisoning

Attacker sends fake ARP replies to associate their MAC with a legitimate IP — enabling man-in-the-middle attacks.

MAC Flooding

Overwhelming a switch's MAC table forces it to behave like a hub — exposing all traffic to all ports.

Packet Analysis

Frame-level inspection in Wireshark requires understanding Ethernet structure — MACs, Type field, FCS.

Asset Identification

OUI lookup on an unknown MAC reveals the device manufacturer — useful for identifying rogue devices.

// Key Takeaways

The OSI model gives us a shared language for troubleshooting and security analysis
L1 handles physical transmission — cables, signals, binary
L2 handles local delivery — NICs, MACs, frames, switches
The Ethernet frame is the fundamental unit of L2 communication
ARP is how devices discover MAC addresses — and a common attack vector
OUI lookup = manufacturer identification, useful in threat hunting and asset management
